# Consistency Check Rules v1.0

## 1. 中心結論

記事内で同一質問への回答が複数ある場合、次の順に確認する。

1. 公式情報または確認可能な根拠
2. 本文で条件を含めて説明された結論
3. 導入・まとめ・FAQの短縮回答

短い回答を優先して本文を上書きしてはならない。

## 2. 数値

### 矛盾とみなす例

- 月200円 / 月500円
- 3つ / 5つ
- 10分 / 30分
- 2025年 / 2026年
- 1GB / 10GB

### 矛盾ではない可能性

- 月200円前後 / 最大500円
- 通常10分 / 初回30分
- 2025年度実績 / 2026年見込み

条件差が明示されている場合は矛盾としない。

## 3. 単位

単位の違いを見落とさない。

例：

- 円/月と円/年
- MBとGB
- 分と時間
- 税込と税別
- 本体価格と総額

数値が同じでも単位が異なる場合は別情報として扱う。

## 4. 年度・日付

次を別々に扱う。

- 公開年
- 更新年
- 対象年度
- 制度施行年
- 商品発売年
- キャンペーン期間

SEOタイトルの年号を変えるだけでは更新完了としない。

## 5. 固有名詞

次を統一する。

- 正式名称
- 略称
- 製品シリーズ名
- OSバージョン
- 人名
- 会社名
- 制度名

ただし、正式名称と一般的な略称の併用は許可する。

初出：

```text
正式名称（略称）
```

以降：

```text
略称
```

## 6. 可否表現

重大矛盾として優先検出する。

- できる / できない
- 対応 / 非対応
- 無料 / 有料
- 必要 / 不要
- 安全 / 危険
- 対象 / 対象外

条件付きの場合は条件を明示する。

## 7. 比較記事

比較軸ごとに対象を固定する。

例：

- 価格
- 容量
- 使用頻度
- 染まり方
- 対応機種

途中で比較対象や評価基準が入れ替わっていないか確認する。

## 8. 手順記事

次を確認する。

- 手順番号の欠落・重複
- 前提操作が後から説明されていないか
- OSや機種による分岐
- 画面名と実際の操作
- FAQの手順が本文と異ならないか

## 9. 条件表現

次の語を追跡する。

- 原則
- 例外
- 一般的に
- 場合による
- 一部
- 最大
- 最低
- 目安
- 約

条件語を削除して断定へ変えない。

## 10. 表記揺れ

low severityの例：

- Wi-Fi / WiFi
- iPhone / アイフォン
- LPLP / ルプルプ
- できる / 出来る

意味が同じなら重大矛盾ではない。

ただし、ブランド表記やSEO上の統一方針がある場合は修正候補とする。
