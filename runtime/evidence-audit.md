# Evidence Audit Runtime Specification v1.0

## 1. 役割

Evidence Auditは、記事内の主張が、その強さと重要性に見合う確認可能性を持つかを評価する。

目的は企業レベルの引用数を増やすことではなく、個人ブログとして読者を誤解させない状態をつくることである。

## 2. 監査対象

- 数値
- 価格
- 年度
- 製品仕様
- 制度
- 法律
- 健康・安全
- 比較優位
- 強い断定
- 体験談
- 推測
- 将来予測

## 3. Evidence Level

### E0 — Personal Experience

本人の体験、感想、使用感。

例：

```text
私の環境では改善しました。
実際に使ってみると軽く感じました。
```

要件：

- 体験であることを明示
- 一般化しない
- 未確認の万人向け結論へ変換しない

### E1 — Commonly Verifiable

一般的で容易に確認できる事実。

例：

- 基本機能
- 一般的な操作
- 広く知られた仕様

要件：

- 誤りがないこと
- 現行情報と矛盾しないこと
- 断定の強さが過剰でないこと

### E2 — Source Recommended

公式情報または信頼できる一次情報の確認を推奨するもの。

例：

- 料金
- 製品仕様
- OS対応
- 制度
- キャンペーン
- 発売日
- サポート条件

### E3 — Source Required

誤情報が読者へ重大な影響を与えるもの。

例：

- 医療
- 法律
- 税務
- 安全
- 金融
- 公的制度の適用条件

## 4. 内部出力

```json
{
  "evidence_audit": {
    "status": "review",
    "score": 78,
    "claims": [
      {
        "text": "必ず電気代を半分にできます。",
        "level": "E2",
        "strength": "strong",
        "support_status": "unsupported",
        "severity": "high",
        "recommended_action": "断定を弱めるか根拠を追加する"
      }
    ],
    "limitations": []
  }
}
```

## 5. Strong Claim判定

次の語は自動禁止ではない。

- 必ず
- 絶対
- 最強
- 一番
- 完全
- 100%
- 誰でも
- 確実
- 唯一
- 最安

確認事項：

1. 比較範囲が明示されているか
2. 根拠があるか
3. 条件があるか
4. 例外がないか
5. 最新性が保たれているか

根拠不足の場合は、次のいずれかとする。

- 条件を加える
- 「可能性があります」等へ弱める
- 体験談として限定する
- 根拠を追加する
- 削除する

## 6. Evidence Score

100点から減点する。

- E3未確認の重大主張：-30
- E2未確認の中心主張：-20
- 強い断定の根拠不足：-15
- 数値の出典・計算根拠不足：-10
- 体験談の一般化：-10
- 軽微な確認不足：-5

最低0点。

## 7. 判定

- 90〜100：pass
- 75〜89：pass_with_suggestions
- 60〜74：review
- 0〜59：fail候補

E3の重大な未確認情報がある場合、点数にかかわらずQuality Gateへblocking issueとして渡す。

## 8. 禁止事項

- 体験談へ不要な出典を要求する。
- 出典がないだけで有用な一般説明を削除する。
- 推測を事実へ変換する。
- 古い公式情報を「公式だから正しい」と扱う。
- 根拠不明の数値を別の推定値で置き換える。
